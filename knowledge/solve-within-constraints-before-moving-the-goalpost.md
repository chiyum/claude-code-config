# 先在現有約束內解題，才准討論「改球門」

**技術域**: 解題方法論 / 流程守則（跨域）
**問題類別**: problem-orientation / goalpost-moving / premature-scope-change / premature-rewrite
**證據等級**: 制度性守則（蒸餾自跨專案 ADR 與知識卡的正反案例；本卡為脫敏通用版）

## 這條守則在防什麼

碰到牆時有兩種第一反應：

- 🟢 **健康**：「以現在的條件，我還能怎麼把球踢進去？」——先窮盡射門角度。
- 🔴 **生病**：「那是不是把球門改大／改一下需求／乾脆重寫？」——第一時間就去動球門。

病**不在「改球門」這個動作本身**（有時球門確實該改）。病在**把「改球門」當成反射性的第一招**，而不是「證明現有條件真的踢不進去之後」才做、且需要 owner 拍板的升級決策。

「改球門」＝任何**放寬約束 / 改需求 / 改凍結門檻 / 縮驗收範圍 / 大重寫 / 造新機制取代硬解**。

## 規則

**想提任何「改球門」方案前，必須先附一段「約束內解題」舉證**——講清楚：在現有條件下我試了什麼、為什麼真的踢不進去。舉不出證據 = 不准提改球門，回頭在約束內解。

舉證至少走過下列一種：

1. **控制變因 / 一個反例**：把問題縮到「只改一個變因」的最小重現，用真環境跑一次。一個反例勝過三輪推理。（真實案例：某除錯連錯三輪根因、還用假測試頁自我循環證成；最後把資料完整鏡射、只改一個欄位、跑真系統，90 秒定案——真根因是一個從沒出現在症狀描述裡的屬性。）
2. **先讀既有能力再說「缺」**：「這系統缺 X」是需要證據的結論，不是預設值——提議造新機制前，先讀完既有 pipeline / 部署腳本 / log。（真實案例：差點為「無法驗證部署版本」補一支新 endpoint，實際上部署腳本一直都會印出當前 commit + 健康檢查，能力一直都在，只是沒人讀 log。）
3. **既有積木拼過了嗎**：重寫 / 抽新層前，先確認 reuse 既有領域積木拼不出來。（真實案例：退款功能沿用既有同步架構而不新增 worker、新流程沿用既有兩段式綁定而不擴充 API——在不破壞既有契約下擴功能。）

## 什麼時候「改球門」是對的（別矯枉過正）

球門確實有該改的時候——但那是**舉證之後的 owner 決策**，不是反射：

- 約束本身就錯了（例：某欄位的計算基準從根本上選錯，該換基準而非硬撐）。
- 需求與現實自相矛盾（例：通用化方案與 1:1 復刻需求不可能同時成立，owner 選了放棄通用性）。

判準：**「賽前就把球門定死、拒絕降級」是紀律的預設**（正面教科書案例：效能門檻在跑出第一個數字之前就寫死，並明列「跑太久就少跑幾年＝禁止」的被否決逃生門）。要動球門，就當成一次明確的、給 owner 看的升級決策——而不是把偵測失敗的能力關掉來讓紅燈消失（「不順的是資料，修那個；不要把 throw 改回盡力而為」）。
