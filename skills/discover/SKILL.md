# /discover — 需求訪談

墊在五步驟的步驟 0 之前。步驟 0 凍結得再嚴，凍結的若是腦補出來的需求，後面全對也是白做——本 skill 先把需求本身釐清，釐清書才交給 PM 凍結。

## 觸發判準（自動偵測）

- **直接開訪談**：需求屬「新產品 / 從零的新服務 / 從零的新頁面群」，且 **≥2 個關鍵維度未指明**（給誰用、核心流程、平台、整合對象等）。例：「幫我做一個航運APP」。
- **先問一句再開**：只命中部分（像大需求但可能使用者已有完整想法）→ 問「這需求維度不少，要不要花幾分鐘訪談釐清？」點頭才開。
- **絕不觸發**：日常 bug fix、小改、既有功能調整、指示已明確的任務——這些照常直接進流程。
- 使用者手動 `/discover <需求>` 一律開，不受判準限制。

## 七大面向（訪談骨架）

| # | 面向 | 在問什麼 | 範例（航運APP） |
|---|---|---|---|
| 1 | 定位 | 這是什麼、給誰用、解決什麼問題 | 形象官網？還是真的要線上賣票？ |
| 2 | 核心流程 | 使用者從進來到完成目標走哪條路 | 查班次→選位→付款→取票？ |
| 3 | 範圍 | 第一版要有什麼、**明確不做什麼** | 會員系統要嗎？退改簽第一版做嗎？ |
| 4 | 視覺 | logo / 品牌色 / 風格參考有無 | 有現成 CI？還是要出三個 direction？ |
| 5 | 技術面 | 平台、與既有系統的關係 | iOS / Android / 兩者？要網頁版嗎？ |
| 6 | 外部整合 | 金流、通知、第三方服務 | 串金流嗎？哪家？要開發票嗎？ |
| 7 | 營運 | 部署、帳號、預估量級、誰維護 | 部署到哪？多少人用？誰管後台？ |

**按任務規模裁剪**：全新產品 → 七大面向全走；既有產品加大功能 → 只問相關面向（定位/視覺通常已知，不重問）；訪談時間與任務大小成比例。

## 提問紀律（讓它幫人而不是煩人）

1. **一次最多 3-4 題、問完一個階段再進下一階段**（用 AskUserQuestion），不把 30 題一次砸過去。
2. **每題附選項＋白話後果＋推薦**：選項寫清楚「選這個對你的操作流程/時程/費用是什麼」（例：「金流：A. 第一版不串、現場付〔最快上線〕(推薦) B. 串〔多 X 工作天＋要申請商店帳號〕」）。不丟開放題逼使用者自己想，用具體選項幫他想。
3. **使用者答「不知道/你決定」→ 給預設值並記入「幫你做的決定」清單**，不卡住。
4. **問之前先自查**：context、repo、產品配置、memory 答得到的不准問（與憲章自答同一精神）；只問「只有使用者知道」的事。
5. **技術題必須翻譯**：術語出現時同句附白話（使用者不必懂 OAuth 是什麼也能回答「要不要讓客人用 LINE 登入」）。

## 終點與產出

**終點判準**：所有「會影響架構的維度」都釘住即收工，不無限追問。

產出**需求釐清書** `~/.claude/acceptance/<YYYYMMDD>-<slug>-discovery.md`：

```markdown
# <需求名> 需求釐清書
## 一句話定位
## 目標使用者與情境
## 核心流程（文字流程圖）
## 第一版範圍（做什麼）
## 明確不做（非目標）
## 視覺方向
## 技術決定（平台/架構/整合）
## 幫你做的決定（使用者答「不知道」時的預設值＋理由）
## 未決事項（不影響開工，開發中再問）
```

寫完向使用者呈現**摘要**確認（不是逐字重講一遍），確認後：

- 接續五步驟 → 釐清書交給 PM 作為步驟 0 凍結驗收清單的依據（原始需求文字照樣附上，釐清書是補充不是取代）。
- 全新產品 → 順手依 CLAUDE.md 規則建 `~/.claude/products/<product>.md`（記得先問 dev/prod 環境區分）。

## 與其他機制的關係

- `/dev` 收到模糊大需求 → 先過本 skill 訪談，再進 /dev 的步驟 0（訪談本身永遠是互動的，即使 /dev auto——需求釐清是使用者唯一不可缺席的一步）。
- 訪談中使用者提的視覺意圖 → 記入釐清書，實作時照 CLAUDE.md 設計鏈規則觸發 ui-designer，不在訪談中處理設計細節。
- 與「約束內解題護欄」互補：護欄防「做的過程中亂改球門」，本 skill 防「一開始就沒弄清楚球門在哪」。
