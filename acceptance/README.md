# acceptance/

存放各任務的凍結驗收清單與驗收證據。

**版控政策**：本 README（制度規則）與各任務清單 `.md` **上版控**；`<任務>/evidence/` 證據目錄（截圖、curl 輸出等大檔）**不上版控**（.gitignore 以 `acceptance/*/` 排除）。制度與清單歷史可同步、可回溯，只有一次性大檔留在本機。

## 檔案結構

- 清單: `<YYYYMMDD>-<任務簡述>.md`（例: `20260703-visitor-reconnect-push.md`）
- 證據: `<YYYYMMDD>-<任務簡述>/evidence/`（與清單同名的目錄；QA/PM 驗收時的截圖與 curl 輸出全存這裡）

## 清單格式（每條三段式）

每個驗收條目必須包含「行為 + 驗證步驟 + 預期結果」，讓任何人（含使用者本人）照表操課就能複驗，不允許驗收者即興決定驗法：

```markdown
### A1 訪客斷線重連後未讀訊息補推
- 驗證步驟: 開 <dev URL>?site=<站台代號> 建訪客房 → 客服（agent01，密碼見 SECRETS.local.md）回 2 則 → 訪客關閉分頁 10 秒後重開同房
- 預期結果: 重連 3 秒內看到那 2 則未讀訊息，順序正確、無重複
```

- 條目編號 `A1`、`A2`… 連號，標題行必須是 `### A<n> ` 開頭（`verify-evidence.sh` 靠此解析）
- 驗證步驟必須寫明: URL / 帳號（引用 SECRETS.local.md，不寫明文）/ 具體操作；驗法在凍結時就定案
- 小改動比例原則不變: 最少 1 條，但仍要三段式

## 證據規約

- QA/PM 每驗一條，至少落地一個證據檔到 `evidence/`，檔名必含 `A<n>-` 段:
  - `qa-A1-重連補推畫面.png`、`qa-A1-messages-curl.txt`、`pm-A2-權限矩陣.png`
- 主 Claude 放行前執行 `bash ~/.claude/scripts/verify-evidence.sh <清單檔>`：任一條目零證據 → 不放行
- 通過腳本後，主 Claude 仍須**親自 Read 抽驗 1-2 張關鍵截圖**，核對畫面內容真的等於條目宣稱；抽驗不符 → 該輪驗收整批作廢重驗

## 任務憲章（大型 / 自主任務加在清單同檔，與清單一起凍結）

```markdown
## 任務憲章（凍結）
- 範圍: <這次要動的>
- 非目標: <明確不做的，防 scope creep>
- 預授權決策:
  - 技術債一律移除，逐項不必確認
  - 命名 / 檔案結構 architect 自行決定
  - 規格模糊處照既有行為 1:1
  - <其他依任務補充>
- 必問白名單: 不可逆刪資料 / 花錢 / 資安 / 碰 prod / 需求自相矛盾（只有這些准中斷）
- 提問政策: 白名單外疑問寫 ~/.claude/state/<task>-questions.md，批次結束一併呈報
```

## 凍結規則

- 產生時機: 開發流程步驟 0，由 PM 依使用者【原始需求】撰寫清單內容，經使用者確認後凍結
- 寫檔分工: PM 沒有寫檔工具，只在回報中【產出清單內容】；由主 Claude 寫入本目錄
- 凍結後（步驟 1~5）任何 agent 不得修改清單本體；`evidence/` 目錄隨時可新增檔案
- 用途: 步驟 2 與步驟 4 的 QA/PM 驗收【唯一依據】。architect 更新的規格書僅作為文件，不作為驗收依據
- 需求中途變更: 由使用者明示後，回到步驟 0 重新產生並確認新清單，舊清單標記為 superseded

> 註：本目錄在公開範本 repo 中只放本 README 當說明；實際任務清單寫在使用者本機的
> `~/.claude/acceptance/`。
